Krav 1 var dessutom det grundkrav jag vart klar med sist just för att jag fastnade ett bra tag på detta krav och insåg att jag inte kunde hänga upp mig på detta krav om jag skulle hinna klart med projektet i tid. Således kopierade jag in den json-struktur från uppgiftsbeskrivningen och satta detta krav på is undertiden som jag genomförde krav 2 och 3.
När de andra kraven var klara tog jag tag i detta krav igen. Jag såg att tidigare redovisningar länkade till en guide för hur en csv till json konvertering kunde gå till men jag ville först försöka lösa kravet helt själv. Tillslut lyckades jag nästan producera den json-struktur som efterfrågades. Mitt problem, som gjorde att jag tillsut använde mig av json-guiden som länkats tidigare, var att hur jag än gjorde innehöll varje representat för "Storlek" en newline som jag aldrig lyckades få bort. Detta gjorde att json-koden inte validerades genom diverse valideringsverktyg. Jag använde mig därför tillslut av guiden och modifierade denna för att passa kravet.
När det slutligen var dags att validera projektet var det bara i princip salar2json.bash som inte validerades och ganska mycket tid spenderades till att tolka valideringsfelen och rätta till dessa.

Överlag var det detta krav som tog mest tid för mig och det känns inte som att jag var direkt förberedd på kravet, det kändes inte som att de tidigare kursmomenten riktigt gav mig kött på benen för att klara av detta krav inom en rimlig tidsram man kan tänka sig att ett första krav för ett projekt ska ta.

Använder verktyget jsonformatter för att validera min genererade jsonfil.

Krav 2: Server för att visa och söka bland salar

Servern bygger på servern från det föregående kursmomentet. När jag började med detta moment skrev jag all kod för servern skulle loopa igenom json-dokumentet och returnera det som skulle direkt i router.get för de olika url'sen. Jag insåg efter att tag att det vore smidigare, mer överskådligt och kontrollerbart att skapa separata funktioner som loopar igenom json-dokumentet och returnerar det som behövs för varje route och sedan anropa funktionerna i router.get och sedan skicka vidare resultatet som funtkionerna returnerar via sendJsonResponse.

När jag väl hade kommit på denna uppdelning och logik i hur en av funktionerna skulle se ut och fungera var det inte så svårt att få till alla funktioner och anropa dessa i rätt router.get.

Den svårate funktionen att få till var den som söker igenom hela json-dokumentet och med hjäl av delsträng returnerar sökningar på alla fält. Jag tolkade detta som att en sökning efter krona och Karlskrona ska visa samma sökresultat. Jag lyckades tillslut lösa detta med att använda mig av Object.keys

/server/index.js innehåller en switch-case där -h, -v och --port går att skicka med som argument. Servern använder sig även av router som kravet specificerade att den skulle göra.

Detta krav kändes som ett rimligt krav och inte alls lika svårt som krav 1 då det kändes som att jag var mer förberedd tack vare tidigare kursmoment att klara av detta krav. Det var helt klart en rimlig deluppgift för projektet i denna kurs enligt mig.

Krav 5: Filtrera antalet träffar

Detta krav gick väldigt fort för mig att implementera då jag hade så mycket kod, funktioner och tankesätt att utgå ifrån för att klara av kravet. Det största problemet jag hade, som egentligen var ett relativt litet sådant var hur query-strängen ?max= skulle sparas ner till en variabel men efter lite googlande kunde jag använda mig av parseInt(req.query.max) för att spara ner filtreringsvärdet. Därefter gjorde jag valet att separera logiken för vad som skulle returneras när ett max-värde var satt som url-parameter och gjorde således separata funktioner som anropades om ett max-värde efterfrågades. Dessa nya funktioner efterliknar de redan skapta funktioner som fanns i /server/server.js med tillägget av en tempIteration variabel och en while-loop. While-loopen ser till att funktionen endast returnerar den mängd resultat som sätts med hjälp av max-värdet.

Jag implementerade även möjligheten att filtrera antalet träffar i klienten. Jag hade först inte tänkt göra det då jag tolkade kravet som sådant att det inte skulle göras, men efter att ha tänkt lite på hur det skulle kunna implementeras insåg att att det inte skulle bli speciellt svårt att låta klienten även bestämma max-värden. Nya funktioner skapades i /client/bthclient.js med max-parametrar som returnerade this.httpGet som motsvarar det som efterfrågas. Därefter skapade jag if-satser i switch-satsen i client/index.js som anropar det nya funktionerna från /client/bthclient.js om rätt args[x] är satt. Meny-funktionen uppdaterades i klienten för att återspegla hur <number> kan användas för att filtrera antalet träffar. Om 'house A-huset 3' skickas in i klienten visas endast 3 träffar för det huset osv.

Detta krav kändes som ett rimligt optionellt krav som byggde vidare på de redan genomförda kraven på ett väldigt bra sätt.

Gillar även alla ssciinema videos som fanns för alla kursmoment även om det var en video som visade helt fel sak (sånt som kan hända i en första version av en kurs).

Rättningarna av kursmomenten har gått något långsamt men jag kan dock inte påstå att det har satt käppar i hjulen för mig på något vis utan det har bara varit att fortsätta med kursmomenten och tänka att "det rättas när det rättas..". Jag tänker mig att rättningshastigheten kanske har något att göra med att kursen är helt ny och försöker ha överseende med detta.

På en skala från 1-10 skulle jag nog ge kursen en 7:a
